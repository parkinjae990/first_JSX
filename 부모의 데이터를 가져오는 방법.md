# 핵심 원리: "열쇠(Key)"만 있으면 문을 열 수 있다

메인 쿼리에서 JOIN을 하지 않아도 되는 이유:: **메인 데이터셋(ds_list)**이 자식 데이터(탭 내용)를 조회하는 데 필요한 **"열쇠(Key 값)"**들을 이미 다 가지고 있기 때문

메인 쿼리(Select A): YEAR(년도), SEMSTR_CD(학기), SBJECT_CD(과목코드), ATNLC_BAN(분반) 등을 조회합니다.

자식 쿼리(Select Detail): 자식 쿼리의 기본키에 해당하는 위 4가지 정보만 알면, 해당 과목의 '수업계획'이나 '평가방법'을 찾을 수 있음.

즉, 메인 쿼리는 내용물(상세 계획)을 다 가져오는 게 아니라, **"이 과목이 누구인지 식별할 수 있는 ID 카드"**만 챙겨온 것

# 쿼리와 스크립트를 통해 설명 


단계: 메인 조회 (DB → 메모리)
사용자가 조회 버튼을 누르면 SELECT ... FROM TH_SUUP_ESTBL_LCTRE A가 실행됩니다.
결과물(년도, 학기, 과목코드 등)이 **ds_list라는 데이터셋(메모리)**에 저장됩니다.
이때는 상세 내용(학습목표, 기자재 등)은 없지만, **식별자(PK)**는 다 있습니다.

2단계: 화면 출력 및 사용자 클릭
그리드에 목록이 뜹니다. 사용자가 **3번째 줄(언어논리... 박제모 교수님)**을 클릭합니다.
넥사크로는 내부적으로 ds_list.rowposition = 2 (0부터 시작하므로)라고 상태를 기억합니다.

3단계: 탭 페이지의 접근 (부모 → 자식 값 전달)
탭 페이지(자식 폼)의 스크립트가 실행될 때(fn_search), 아래 코드가 돕니다.

``
code
JavaScript
// "부모님(fv_parentObj), ds_list 데이터셋 좀 보여주세요."
// "현재 선택된 줄(rowposition)의 'year' 값이 뭡니까?"
var lv_year = fv_parentObj.ds_list.getColumn(fv_parentObj.ds_list.rowposition, "year");
이 코드는 DB에 다시 다녀오는 것이 아닙니다.
아까 1단계에서 이미 가져와서 브라우저 메모리에 들고 있던 값을 끄집어내는 것입니다.
4단계: 상세 조회 (자식 → DB)
이제 자식 폼은 부모에게서 얻은 **열쇠(년도, 학기, 과목코드)**를 가지고 자신만의 쿼리를 서버로 보냅니다.
code
``


SQL
-- 자식 폼용 쿼리 (예: 수업계획 탭)
SELECT * FROM 수업계획테이블
WHERE YEAR = '2025' AND SBJECT_CD = '12345' ...
DB는 이 조건에 맞는 상세 데이터만 딱 1건 찾아서 탭 페이지에 돌려줍니다.


3. 비유로 이해하기 (호텔 프론트)
메인 쿼리 (프론트 데스크):
투숙객 명단(ds_list)을 가지고 있습니다.
명단에는 **방 번호(Key)**와 이름만 있습니다. (손님이 저녁으로 뭘 먹었는지는 안 적혀 있음)
프론트 직원은 옆에 있는 식당 장부와 **조인(Join)**해서 모든 정보를 한 번에 보지 않습니다. 그냥 명단만 봅니다.
탭 페이지 (룸서비스 직원):
룸서비스 직원은 손님이 호출하면 프론트에 묻습니다. "지금 선택된 방(rowposition) 번호가 몇 번입니까?"
프론트 직원은 **장부(메모리)**를 보고 "101호입니다"라고 알려줍니다. (fv_parentObj...getColumn)
룸서비스 직원은 "101호"라는 정보(파라미터)만 가지고 주방(DB)에 가서 101호 주문 내역을 가져옵니다.



# 요약
메인 쿼리는 자식 데이터를 몰라도 됩니다. 자식 데이터를 찾을 수 있는 **ID(Key)**만 있으면 됩니다.
넥사크로 스크립트는 DB가 아니라 **이미 조회된 화면의 데이터셋(메모리)**을 참조하는 것입니다.
따라서 메인 쿼리에 조인을 걸어서 데이터를 뻥튀기할 필요 없이, 클릭한 순간의 PK 값만 넘겨주면 자식 탭이 알아서 조회를 할 수 있는 구조가 됩니다.
